## 0. 最关键的前置说明（请先读）

### 0.1 两版是两套独立系统、两个独立页面

公司目前有两套**完全独立**的系统：

*   **美工版**（供设计部使用）

*   **运营版**（供深圳运营、成都运营共用）


「查看已完成生图」在两套系统里**各有一个独立页面**：独立部署、独立路由、独立前端代码仓、独立后端服务（或至少独立接口路径）。两版差异**非常小且收敛**（见第 4 章），但这不是合并成一个带切换器的单页的理由，请不要这么做。两版的用户群、数据范围、权限规则都不同，合并会导致权限边界不清晰、代码后续难以维护。

**请严格按两套独立页面实现。**

**页面入口**：在【新版AI生图】下继续增加子菜单【美工版查看已完成生图】和【运营版查看已完成生图】

## 1. 背景与目标

### 1.1 业务背景

公司的 AI 生图工作台（Tab1~Tab5）已支持任务创建、生图、审核、推送全链路。当一批图片进入「已完成」状态后，用户需要一个**独立的回看/查询页面**，用于：

*   按任务、按图片维度多条件筛选，快速找到需要的已完成图片

*   打开沉浸式大屏查看 L3 弹窗，对单张图做详细查看


本页面不承担**创建、编辑、审核、推送**等产生状态变化的能力，是一个**只读查询页**。

### 1.2 目标用户

| 系统 | 用户群 |
| --- | --- |
| 美工版 | 设计部的专员、组主管、子部门高级主管、设计总监 |
| 运营版 | 深圳运营、成都运营的专员、组主管、部门高级主管、运营总监，以及特殊团队成员（业务管理部、总监助理、FBA/FBM 策略管理组等） |

其他跨系统角色（产品经理、研发、测试、技术支持）作为管理员可见两版全部数据，详见第 5 章。

### 1.3 产品目标

*   让用户**快速找到**过往已完成的生图结果（主要场景：回查某 SKU / 商品的历史生图、对比不同模型产出、批量整理需要推送给下游的图片集合）

*   严格遵循**数据权限**，不让用户看到越权数据

*   与沉浸式大屏查看 L3 **无缝衔接**，点一下图直接进入精细查看

## 5. 数据权限（整个需求的安全核心，必读）

### 5.1 总则

*   公司有两套**互相独立**的系统：美工版、运营版（运营版由深圳+成都共用）

*   **运营版原则**：组与组之间、部门与部门之间**互不可见（赛马竞争）**。每个人默认只能看到自己管辖范围内的数据

*   **美工版原则**：组与组、部门与部门之间**互相协作**，不做屏蔽，主管及以上可看全体


### 5.2 运营版数据权限

#### 5.2.1 深圳运营

**按岗位层级，每个人默认能看到的范围：**

*   专员 → 自己的数据

*   组主管（A1 组、A2 组……）→ 本组全体

*   部门高级主管（A、B、C、D、E、F、日本、精品、爆旺 共 9 个部门）→ 本部门全体

*   深圳运营总监 → 管理员权限，可见全司运营（深圳 + 成都）；如业务需要，可额外授予美工版管理员权限以查看全司美工


**特殊组织结构：** 精品、爆旺这两个部门组织结构扁平，没有「组」这一层，专员直接向部门高级主管汇报。

**例外人群（不受上述层级限制，有更大的可见范围）：**

*   **业务管理部全体成员**（经营分析组、项目管理组、数据分析组）以及**深圳总监助理**（多位），哪怕岗位是专员 → 仅可见深圳运营全体，不可见成都


#### 5.2.2 成都运营

**按岗位层级，每个人默认能看到的范围：**

*   专员 → 自己的数据

*   组主管（1 组、2 组……）→ 本组全体

*   部门高级主管（FBA、FBM 共 2 个部门）→ 本部门全体

*   成都运营总监 → 管理员权限，可见全司运营（深圳 + 成都）；如业务需要，可额外授予美工版管理员权限以查看全司美工


**例外人群：**

*   **FBA 策略管理组全体成员**，哪怕岗位是专员 → 可见 FBA 部门全体

*   **FBM 策略管理组全体成员**，哪怕岗位是专员 → 可见 FBM 部门全体


### 5.3 美工版数据权限

美工条线协作互通，整体放宽：

*   设计部下设**精品设计、铺货设计**两个子部门，各子部门下设若干组

*   专员 → 自己的数据

*   所有组长主管、子部门高级主管 → **管理员权限，可见美工全体**

*   设计总监 → 管理员权限，可见美工全体；如业务需要，可额外授予运营版管理员权限以查看全司运营


### 5.4 公共管理角色（跨系统）

以下人员因承担产品建设与运维职责，对**运营版、美工版两套系统分别**拥有管理员权限（看全体）：

*   产品经理

*   本项目前端研发

*   本项目后端研发

*   本项目测试

*   技术支持


### 5.5 三条业务规则归纳（实现时的判定顺序）

后端在计算「当前用户的可见范围」时，按以下顺序判定：

**规则 1：系统级管理员（优先级最高）**

若当前用户属于以下任一角色，直接返回「全系统可见」：

*   运营版管理员：深圳运营总监、成都运营总监、产品经理、前后端研发、测试、技术支持

*   美工版管理员：设计总监、所有美工组长主管、所有美工子部门高级主管、产品经理、前后端研发、测试、技术支持

*   跨系统授权：运营总监可按需额外获得美工版管理员权限；设计总监可按需额外获得运营版管理员权限（这一类授权通过配置表维护）


**规则 2：特殊团队/特殊人员整体上提**

若规则 1 不命中，检查用户是否属于以下特殊集合，命中则返回对应的上提范围：

| 特殊集合 | 可见范围 |
| --- | --- |
| 深圳业务管理部全体成员（经营分析组、项目管理组、数据分析组） | 深圳运营全体 |
| 深圳总监助理（多位） | 深圳运营全体 |
| 成都 FBA 策略管理组全体成员 | FBA 部门全体 |
| 成都 FBM 策略管理组全体成员 | FBM 部门全体 |

**规则 3：默认层级规则（兜底）**

若前两条都不命中，按岗位层级判定：

*   运营条线（严格）：

    *   专员 → 自己

    *   组主管 → 本组

    *   部门高级主管 → 本部门

*   美工条线（宽松）：

    *   专员 → 自己

    *   组主管、子部门高级主管 → 本系统全体


### 5.6 数据权限如何落地到本页面

**两件事：**

#### （a）创建人筛选下拉的可选范围

下拉树展示的人 = 当前登录用户在**本系统**下的可见范围。

**举例：**

| 登录用户 | 登录系统 | 创建人下拉里能看到 |
| --- | --- | --- |
| 深圳运营 A 部门某专员 | 运营版 | 只看到自己 |
| 深圳运营 A 部门 A1 组主管 | 运营版 | A1 组全体（含自己） |
| 深圳运营 A 部门高级主管 | 运营版 | A 部门所有组的所有成员（含自己） |
| 深圳总监 | 运营版 | 深圳运营 + 成都运营全体 |
| 美工版某专员 | 美工版 | 只看到自己 |
| 美工版某组主管 | 美工版 | 美工版全体 |
| 产品经理 | 美工版 | 美工版全体 |
| 产品经理 | 运营版 | 运营版全体 |
| 深圳业务管理部某成员 | 运营版 | 深圳运营全体 |

**默认选中：** 进入页面时，下拉里的所有人**默认全选**（注意：是"默认全选范围内的全部"，不是"整个公司")。

**边界：** 用户可以手动取消部分选择，但至少保留 1 人（前端强制，全选取消到 0 时弹 toast 阻止）。

#### （b）主列表数据也受可见范围约束（后端强约束）

即使前端请求里没传 `creatorIds`，后端也**必须**只返回可见范围内的任务/图片。

即使前端传的 `creatorIds` 里包含越界 ID（理论上只会在接口被直接调用时发生），后端也**必须**返回 400 错误，不返回列表数据。

**这是后端兜底约束，不能依赖前端约束来保证安全。**

### 5.7 两系统独立不共享

*   用户登录态不共享：美工版、运营版各自一套登录

*   但一个真实物理人可能同时在两个系统里有账号（例如设计总监在美工版有设计总监身份，在运营版可能有运营版管理员授权），这两个账号各走各的权限体系

*   数据不共享：美工版的任务/图片在运营版查询不到，反之亦然

## 6. 功能详述

### 6.1 筛选的两个作用域（核心概念）

筛选项分两类，视觉上用「任务」**绿标签** / 「图片」**蓝标签**分行标记（详见原型筛选区）：

**任务维度筛选**（命中即整个任务连同其所有图片从列表移除）：

*   创建人

*   时间范围

*   关键字搜索（美工版=SKU/TaskID；运营版=店铺/ASIN/TaskID）

*   张数数值筛选（例如"通过张数 ≥ 5 的任务"）


**图片维度筛选**（命中即仅该张图片从列表剔除，其所在任务保留）：

*   模型

*   图片类型

*   侵权禁售状态

*   状态子 Tab（全部图片 / 通过图片 / 不通过图片）


**重要行为**：一个任务下所有图片都被图片维度筛选剔除后，该任务也会从列表移除（因为空任务无展示价值）。分页的「共 N 个任务」的 N 是**过滤后仍有可见图片的任务数**。

### 6.2 筛选项逐一详述

#### 6.2.1 筛选生效方式：即时 vs. pending

| 筛选项 | 生效方式 |
| --- | --- |
| 状态子 Tab（全部图片 / 通过图片 / 不通过图片） | **即时** |
| 排序维度、升降序 | **即时** |
| 展示密度 | **即时** |
| 创建人 | pending，需点「查询」 |
| 时间范围 | pending，需点「查询」 |
| 关键字搜索 | pending，需点「查询」（回车等同于点查询） |
| 模型 / 图片类型 / 侵权禁售 | pending，需点「查询」 |
| 张数数值筛选 | pending，需点「查询」 |

**「查询」按钮交互**：

*   当用户改了任何一个 pending 项、但还没点查询时，按钮变为 dirty 态（绿色加深 + 右上角圆点提示）

*   点查询后：pending 复制到 filters，重置到第 1 页，重新渲染，dirty 态清除


**「重置」按钮**：

*   清空所有筛选条件回到默认值（创建人全选当前可见范围、时间=**创建时间 · 最近 7 天**、其余全为空）

*   状态子 Tab 回到「全部图片」

*   页码回到第 1 页

*   不清空排序、密度（这两项属于视图偏好）


#### 6.2.2 创建人筛选

*   **形态**：触发器（显示已选 chips）+ 下拉树（部门 → 组 → 专员 三层）

*   **范围**：按 5.6(a) 计算

*   **交互细节**：

    *   三态 checkbox（全选/半选/未选）逐层联动

    *   对精品、爆旺等没有「组」层级的扁平部门，人员直接挂在部门下展示，不额外生成「未分组」

    *   部门/组全选时显示为聚合 chip（如「A 部门（24 人）」），避免 chip 爆炸

    *   顶部搜索框按姓名过滤（大小写不敏感），命中项所在节点自动展开

    *   当前登录用户旁显示蓝色「我」标签

    *   默认展开当前用户所在部门，其他部门收起

    *   全选取消到 0 时阻止并 toast（「至少保留一位创建人」）


#### 6.2.3 时间筛选

*   **触发器**：显示当前维度 + 当前选中的预设（例如「创建时间 · 最近 7 天」）

*   **面板内容**：

    *   维度 Tab：创建时间 / 更新时间

    *   快捷预设：最近 7 天（默认）/ 最近 30 天 / 本周 / 本月

    *   自定义：起止日期选择器

*   **作用域**：任务维度，按 `Task.createdAt` / `Task.updatedAt` 过滤任务，不按单张图片时间过滤

*   **行为**：在面板内选择维度或预设时，面板 UI 立即反映选中态，同时写入 pending（「查询」按钮转 dirty）；点「查询」后列表才真的刷新。面板没有独立的「应用」按钮


#### 6.2.4 关键字搜索

*   **仅支持精确匹配**：关键字必须与目标字段**完整相等**才命中；**不支持**模糊 / 部分 / 前后缀匹配

    *   示例：搜索 `22062564` 会命中 `sku === '22062564'` 的记录；搜索 `22062` 不会命中 `22062564`

*   大小写不敏感，前后空格自动去除

*   美工版：匹配 系统SKU **或** TaskID

*   运营版：匹配 店铺名 **或** ASIN **或** TaskID

*   空字符串 = 不过滤

*   回车等同于点「查询」


#### 6.2.5 模型筛选

下拉单选：

*   v1 选项：全部模型 / LinkFox / 自研 / 阿里绘蛙

#### 6.2.6 图片类型筛选

下拉单选：

*   美工版 v1、运营版 v1 一致：全部类型 / 场景图 / 特写图-白底 / 特写图-非白底 / 卖点图 / 标准普通 A+ 图


#### 6.2.7 侵权禁售状态筛选

下拉单选：全部 / 通过 / 不通过 / 未检测

**业务规则（美工版 v1、运营版 v1 一致）：**

*   仅「卖点图」和「标准普通 A+ 图」会进入侵权检测

*   其他图片类型（场景图、特写-白底、特写-非白底）按业务规则**恒为「未检测」**（不是没检测到，是按规则不进检测）

*   被检测的图，结果为「通过」或「不通过」


#### 6.2.8 张数数值筛选

组合控件：维度下拉 + 操作符下拉 + 数值输入框

*   维度：审核通过张数 / 审核不通过张数

*   操作符：≥ / ≤ / =

*   数值：正整数，空值 = 不过滤

*   作用域：任务维度（例：「审核通过张数 ≥ 5」会过滤掉审核通过张数小于 5 的整个任务）


### 6.3 状态子 Tab

位置：筛选区上方，紧跟页头

选项：全部图片 / 通过图片 / 不通过图片

*   每个 Tab 右侧带数字角标 = 该状态下当前可见范围内的图片数量（注意是**当前筛选后、未经状态过滤**的基数）

*   选中「全部图片」：不按状态过滤

*   选中「通过图片」：只保留 `status === 'approved'` 的图

*   选中「不通过图片」：只保留 `status === 'rejected'` 的图

*   「通过 / 不通过」指**人工审核状态**，与侵权禁售检测状态是两回事（两条独立的状态维度）


### 6.4 排序

*   维度：创建时间（默认）/ 更新时间

*   顺序：默认降序，可切换升序

*   即时生效

*   排序对象是任务，按 `Task.createdAt` / `Task.updatedAt` 排序；任务内图片再按图片类型顺序 + 同类型序号展示


### 6.5 展示密度

三档（即时）：

*   紧凑（最小 120px 列宽）

*   标准（180px，默认）

*   大图（260px）


实现上是 `grid-template-columns: repeat(auto-fill, minmax(Xpx, 1fr))`，响应式自动计算每行列数。

### 6.6 任务分隔条

每个任务上方一条双行分隔条：

**第一行**（左→右）：

*   TaskID（加粗，例 `T20260324_000001`）

*   身份字段（美工版：`系统SKU: {sku}`；运营版：`店铺: {storeName}` + `ASIN: {asin}`）

*   标题（单行省略，hover 显示完整 tooltip）

*   右侧：统计 

    *   这里的统计是基于**任务真实全量图片**，不受当前图片维度筛选影响

    *   即使用户筛选了"只看卖点图"，这里的"共 32"仍然显示任务总共 32 张图，而不是 32 张里的卖点图数


**第二行**（左→右）：

*   创建人（姓名）

*   创建时间

*   更新时间

*   商品尺寸（`{长} × {宽} × {高} cm`）


### 6.7 图片卡片

**卡片内容：**

*   缩略图（占主体）

*   右上角：模型徽章（短码，按模型有颜色区分）+ 审核状态徽章（✓ / ✕，绿色/红色）

*   底部标签：

    *   序号是任务内同类型图片的递增序号，从 1 开始


**交互：**

*   Hover：出现半透明黑色蒙层 + 居中单颗按钮「沉浸大屏」（带 `fa-expand` 图标）

*   点击卡片任意位置 **或** 点击按钮 → 打开沉浸式大屏查看弹窗（见 6.8）

*   整张卡 + hover 按钮都是同一个入口，幂等


### 6.8 沉浸式大屏查看（重要：复用，全屏弹窗）

**本页不实现沉浸大屏弹窗**。

沉浸式大屏查看（L3）是公司**已上线、已成熟**的既有产品功能，以**全屏模态弹窗**的形式覆盖在当前页面之上，用于对单张图片做精细查看（含左参考右 AI 生成对比、缩略图切换带、右侧属性面板、缩放/拖动、商品文本信息展示等）。

**关键交互模型：** 点击图片后是在当前列表页**之上打开一个弹窗**，不是路由跳转、不是新开页面。关闭弹窗后用户仍在原来的列表页、原来的滚动位置、原来的筛选条件上。

**对接方式：**

*   原型里的 `openImmersiveL3(photoId)` 是桩函数，仅 toast 提示

*   研发实现时，替换为真实调用（打开全屏模态组件），例如：具体调用方式以公司内部 L3 组件的实际形态为准

*   美工版、运营版打开的是**同一个 L3 组件**，只是传入的 photoId / taskId 来自各自系统的数据源，两版代码实现完全一致


**不要重绘沉浸大屏弹窗。**

### 6.9 分页器

位置：页面底部常驻

*   **单位：按任务分页**（不是按图片）

*   每页条数：10 / 20（默认）/ 50

*   显示：`共 N 个任务 · 第 M / T 页` + 范围 `X-Y` + 每页选择器

*   按钮：首页 / 上页 / 页码按钮（当前页两侧各 1 个，加首末页和省略号）/ 下页 / 末页 / 跳转输入框

*   「共 N 个任务」的 N 是**所有筛选（任务维度 + 图片维度 + 状态 Tab）应用后**仍然有可见图片的任务数

*   一个任务下所有可见图片一起展示在同一页，不跨页拆分

*   切换页码：滚动到页面顶部（smooth）


### 6.10 空态

触发条件：筛选后一个任务都没有

表现：主内容区显示空态（具体视觉见原型，目前是极简灰色文案）

### 6.11 Toast

出现场景：

*   创建人全选被取消到 0 → info: 「至少保留一位创建人」

*   （本页其他场景暂无需要 toast 的操作）


样式位置：页面顶部居中，1.8s 自动消失

## 8. 接口契约（建议骨架）

> 以下仅为建议，字段名/路径/传参风格由后端自行决定，只要语义等价即可。

### 8.1 获取当前用户可见的创建人树

```plaintext
GET /api/completed-images/visible-creators

返回:
{
  currentUser: { id, name, deptId, groupId },
  tree: [
    {
      deptId: "dept-a",
      deptName: "A 部门",
      users: [{ id, name, short, color }, ...],  // 无组部门人员直接挂在部门下，可为空
      groups: [
        {
          groupId: "grp-a1",
          groupName: "A1 组",
          users: [{ id, name, short, color }, ...]
        },
        ...
      ]
    },
    ...
  ]
}

```

后端已经应用了 5.5 的三条规则，返回的 tree 就是可见范围；前端直接渲染。

### 8.2 查询已完成图片列表

```plaintext
GET /api/completed-images/list
  ?page=1
  &pageSize=20                  // 10 | 20 | 50
  &sortBy=created               // created | updated
  &sortOrder=desc               // desc | asc
  &statusTab=all                // all | approved | rejected
  &creatorIds=u1,u2,u3          // 逗号分隔；为空 = 可见范围全选；包含越权 ID 时返回 400
  &timeDim=created              // created | updated
  &timeFrom=2026-04-17
  &timeTo=2026-04-24
  &keyword=22062564             // 空 = 不过滤
  &modelId=linkfox              // 空 = 不过滤
  &imageType=scene              // 空 = 不过滤
  &infringementStatus=passed    // passed | rejected | untested | 空
  &countDim=approved            // approved | rejected
  &countOp=gte                  // gte | lte | eq
  &countVal=5                   // 空 = 不过滤

返回:
{
  page: 1,
  pageSize: 20,
  totalTasks: 137,              // 所有筛选应用后的任务总数（分页基数）
  statusCounts: {               // 用于子 Tab 数字角标（经所有筛选、未经 statusTab 过滤）
    all: 1284,
    approved: 1032,
    rejected: 252
  },
  tasks: [
    {
      id: "T20260324_000001",
      sku: "22062564",
      storeName: "Amazon-Z01466-UK",    // 美工版可以不返回或为 null
      asin: "B08DNRKJX7",                // 美工版可以不返回或为 null
      title: "...",
      owner: { id, name },
      createdAt, updatedAt,
      size: { length, width, height },
      counts: {                 // 基于任务真实全量，不受筛选影响
        total: 32,
        approved: 28,
        rejected: 4
      },
      photos: [                 // 已经应用所有图片维度筛选 + statusTab
        {
          id, type, typeIndex, model,
          status, pushed,
          infringement: { status, detail },
          imgUrl
        },
        ...
      ]
    },
    ...
  ]
}

```

**后端关键职责：**

1.  先按当前用户身份计算可见范围（规则 1→2→3）

2.  校验 `creatorIds` 是可见范围子集；如包含越界 ID，直接返回 400

3.  在可见范围内应用任务维度筛选 → 得到候选任务集

4.  对每个候选任务，先应用除 `statusTab` 外的图片维度筛选（模型 / 图片类型 / 侵权禁售）→ 得到 `statusCounts` 的统计基数

5.  在上述基数上应用 `statusTab` → 过滤其 `photos` 数组

6.  剔除 photos 为空的任务

7.  按任务创建/更新时间排序、按任务分页、返回


`statusCounts` 要注意：它是"不考虑 statusTab 的情况下各状态的数量"，用于驱动前端子 Tab 角标。实现时应在已应用任务维度筛选、且已应用模型/图片类型/侵权禁售筛选后，再按人工审核状态分别统计。

### 8.3 沉浸大屏入口

本页面不调用专门的"打开沉浸大屏"接口；前端点击卡片后**直接调用公司内部已有的 L3 组件**打开（或走 L3 的路由）。参数需要 `photoId`、`taskId`，以及只读标记（该页进入 L3 应为只读态，用户不应在回看页触发审核变更）。

## 9. 边界情况与错误处理

| 场景 | 期望行为 |
| --- | --- |
| 当前用户无可见数据（权限范围为空或范围内无已完成任务） | 列表空态；创建人下拉只含自己 |
| 筛选组合无结果 | 列表空态，筛选条件保留，用户可手动回退 |
| 创建人尝试清空到 0 | 前端阻止，toast 提示"至少保留一位创建人" |
| 分页跳转到超界页 | 自动回退到最后一页 |
| 直接访问越权 taskId 的深链（如果有） | 后端 403 / 404 |
| 网络错误 / 接口超时 | 列表区错误态 + 重试按钮（具体样式见前端通用错误组件） |
| 任务下所有图片被图片筛选剔除 | 该任务整个从列表消失 |
| 任务数据有但 `finishedAt` 缺失 | 不应出现（本页只查 completed，必有 finishedAt）；若出现视为脏数据，前端容错跳过 |
